- (decision) AD-1 Replay reuses live domain shape. Binds: FR-5-FR-9 (Race Replay), live-race components (GapList, SectorBoard, LapTimeChart, EventTimeline, TrackMap). Prevents: replay path and live path rendering from diverging into two component trees / two data shapes. Rule: replay produces the same RaceStateSnapshot/DriverState-shaped frames as live SignalR does, normalized through the existing normalizeSnapshot() before hitting Zustand — every live-race component stays source-agnostic (no isReplay branching inside GapList/SectorBoard/etc).
- (decision) AD-2 Replay data fetch is a new bounded backend endpoint, not live streaming replayed. Binds: FR-5, FR-6-FR-9. Prevents: a dev wiring replay through RaceHub/SignalR (implies a fake live session, contradicts NFR 'no new live-data source'). Rule: backend adds GET /api/races/{season}/{round}/replay returning the full ordered array of per-lap RaceStateSnapshot frames, built by RaceReplayService from the same OpenF1 historical REST + Ergast sources as the existing fallback-to-last-race path (MVP FR-16). Cached IMemoryCache, 7-day TTL tier (immutable once race completed, same tier as historical race results). Frontend fetches this once via TanStack Query, not per-lap.
- (decision) AD-3 Replay state split follows the existing TanStack Query / Zustand boundary, extended not violated. Binds: FR-6-FR-9, liveRaceStore hard rule (Zustand never holds server data). Prevents: the fetched replay frame array being stuffed into Zustand (violates existing hard rule), or playback position being re-fetched per lap. Rule: the frame array from AD-2's endpoint lives only in TanStack Query cache; a new replayStore (Zustand) holds only UI-interaction state — currentLapIndex, isPlaying, speed (1|2|4) — and derives the displayed frame by indexing into the query's cached array. Scrub = direct index set. No network call per lap/scrub.
- (decision) AD-4 Replay ticking is client-side only. Binds: FR-6-FR-9. Prevents: a second real-time channel (SignalR subscription or polling loop) being built for replay, duplicating RaceDataOrchestrator's job for historical data. Rule: playback advances currentLapIndex via a client-side interval timer at (baseIntervalMs / speed); pause clears the timer; scrub sets the index directly with no animation catch-up, matching EXPERIENCE.md's snap-to-lap contract.
- (decision) AD-5 Circuit outline reuses and extends the existing circuit-configs asset, not a second asset class. Binds: FR-4, FR-13, existing TrackMap.tsx/circuit-configs pattern. Prevents: a parallel 'decorative outline' asset pipeline coexisting with the live-map's trackPath+transform config (two sources of truth for the same shape). Rule: circuit-configs/{circuitId}.json's existing trackPath field (already viewBox-scoped SVG path data, already used by TrackMap.tsx) is the single source for every rendering of a circuit's shape — calendar card (FR-4, small crop), Race Weekend Detail (FR-13, large crop), and the live map overlay (existing) all consume the same file at different CSS sizes. Coverage expands from 1 circuit (Monza only, phase-1 POC scope) to the full current-season calendar, sourced from f1db/f1db (CC-BY-4.0) using each circuit's '-present' layout (e.g. monza-7, not monza-1).

- (decision) AD-9 Fan Card storage migrates to a keyed collection under a new versioned localStorage key. Binds: FR-12, existing useFanCardStore/localStorage versioning convention (f1app__{feature}__{key}). - (decision) AD-10 [ASSUMPTION] Fan Card asset sourcing (FR-11) resolves the addendum's open question: manual curation, no new external API. Binds: FR-11.
